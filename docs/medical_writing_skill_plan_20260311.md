## 零、对话背景与讨论前提

本方案不是从空白开始提出的，而是基于前两轮讨论逐步收敛出来的。

为方便其他 AI 复核，这里先把讨论前提写清楚：

### 0.1 第一轮讨论的核心问题

第一轮不是直接讨论“怎么重写 skill”，而是先判断：

> 当前这套医学写作 skill，到底是不是已经过于定式化？

在这一轮里，形成了两个关键判断：

1. 当前系统确实已经有一套经过实战验证的医学写作方法。
2. 问题不在于“规则太多”，而在于“把必须稳定的硬约束”和“本来应该可变的写法”绑得太死。

也就是说，当时的讨论结论不是“整套推翻”，而是：

- 保留稳定的约束底盘
- 调整过于统一的写法骨架
- 让不同文章不要都长成同一张脸

### 0.2 第二轮讨论的核心问题

第二轮不是继续讨论“定式对不对”，而是进一步讨论：

> 既然已经有一套能打的写作方法，是否应该把它包装成一个独立的医学写作类目，再由更灵活的广义 skill 决定什么时候触发它？

在这一轮里，形成了新的方向性共识：

1. 当前这套医学写作方法，已经接近一套“经过考验可以上菜单的菜”。
2. 不应该让每次请求都由执行者重新临场判断“这道菜到底怎么做”。
3. 更合理的结构是：
   - 外层由更广义的入口负责分流
   - 内层由单独封装的医学写作类目负责稳定出品

这也是本方案最终采用“灵活触发层 + 稳定写作类目层 + 子模块层”结构的直接来源。

### 0.3 本方案的边界

因此，本方案解决的不是“医学写作是否存在”，而是下面这三个更具体的问题：

1. 当前已经存在的医学写作能力，哪些部分应继续保留为稳定底盘。
2. 哪些部分应从硬规则降为可选原型或可变策略。
3. 上游触发层应如何改造，才能让这套能力更容易被正确调用，而不是继续依赖执行者临场匹配。

### 0.4 复核时应注意区分两类信息

本方案同时包含两类来源：

- **仓库可核对事实**
  - 上游 `opencode` 当前有哪些医学相关 skill
  - 仓库里保留的旧医学写作核心长什么样
  - 当前 V2 技能入口如何说明这些 skill 的关系

- **本轮设计讨论形成的共识**
  - “这套方法已经是一道成熟菜”
  - “不应让厨师每次临场单独匹配”
  - “应采用外层分流、内层稳定出品的封装方式”

前者可以直接用文件核对，后者属于设计判断，应按架构合理性复核，而不是误当成仓库现状。

## 一、背景

当前上游 `opencode` 已经有一组医学相关 skill：

- `medical-writing`
- `medical-opinion-generator`
- `medical-article-reviewer`
- `medical-database-query`

但这组 skill 的实际分工并不对称：

- 上游 `medical-writing` 相对较薄，主要负责“什么时候触发”、运行规则入口和组合请求转接。
- 上游 `medical-article-reviewer` 与 `medical-database-query` 并不是纯薄壳，它们已经内含 `M2` 叙事检查、`L3/L4` 数据结构、质量门和对接约束。
- 真正让文章变得稳定、可控、同时也显得有些定式的部分，主要仍沉在历史医学写作核心规则里。

这意味着现在的问题不是“没有一套能用的菜谱”，而是：

1. 已经有一套相对成熟的菜谱，但菜谱里的写法约束和安全约束绑得太紧。
2. 当前触发层过薄，遇到稍微模糊一点的用户意图，就容易又回到“由执行者临场猜该走哪条链路”。
3. 结果是同一种医学内容会越来越稳，但也越来越像同一张脸。

你现在提出的方向是合理的：

> 把当前已经验证过的一套医学写作方法，略作去定式化调整后，封装成一个稳定可复用的“医学写作类目”；再由更灵活的广义 skill 去决定什么时候触发它。

这相当于把系统从“厨师临场判断每道菜怎么做”，调整成“菜单先定义一个可靠菜系，再由前厅判断客人应该点哪道菜”。

## 二、当前问题定位

### 2.1 目前真正过于定式的，不是触发壳，而是写作策略层

当前上游 `medical-writing` skill 的职责其实比较干净：

- 定义触发语义
- 读取运行规则
- 启动日志
- 把具体规则委托给知识库

它本身并没有写死太多文章结构细节。

真正导致“系统味”偏强的，是旧医学写作核心里这些策略被写得过硬：

- 统一总线顺序固定
- 统一叙事骨架过强
- “一文一主张”适用范围过宽
- 段落功能块绑定过死
- 审核器也用同一骨架反向检查

所以问题不应该被描述成“上游 skill 写坏了”，而应更准确地理解为：

> 触发层已经很轻，但下游被调用的写作策略层，仍然像一个高约束编译器，而不是一个有弹性的写作系统。

### 2.2 当前还缺一个真正的医学内容总入口

现在上游的 4 个医学 skill 是并列关系：

- 写作
- 观点
- 审核
- 查询

这组并列关系适合熟悉系统的人手动点选，但不适合更广义、更模糊的真实请求。

比如下面这些请求，当前都容易让执行者先自己猜：

- “帮我把这些文献做成一篇可交付文章”
- “先看证据，再形成一个可写的稿子”
- “这批素材到底适合做观点稿还是综述稿”
- “我想做一套系列稿，先帮我定每篇怎么写”

这说明系统已经有“菜”，但还缺一个真正的“前厅分流”。

## 三、建议的封装方向

### 3.1 整体结构

建议把医学内容能力拆成三层：

1. **广义触发层**
2. **医学写作类目层**
3. **具体能力子模块层**

对应关系如下：

| 层级 | 职责 | 是否需要灵活 | 建议状态 |
|------|------|--------------|----------|
| 广义触发层 | 判断用户到底要查询、观点、写作、审核还是系列策划，并选择固定链路模板 | 高 | 新增 `medical-content-router` |
| 医学写作类目层 | 提供稳定可复用的医学写作主链与原型接口 | 中 | 保留并重构现有 `medical-writing` |
| 子模块层 | 查询、观点、审核、证据策划、系列规划等专项能力 | 中 | 复用并补齐 |

### 3.2 一句话原则

> 让“何时触发”变灵活，让“触发后怎么稳稳写出来”保持稳定。

### 3.3 唯一 live 落点与执行权威

在进入实施设计前，先把“哪一个真实 skill 文件承接哪一层”钉死，避免出现双入口并存：

| 层级 | 唯一 live 文件 | 非 live 参考文件 | 角色说明 | 退役/保留策略 |
|------|----------------|------------------|----------|---------------|
| 广义触发层 | `C:\Users\96138\.config\opencode\skills\medical-content-router\SKILL.md` | 本方案文档、`medical_kb_system_v2/skills/README.md` | 唯一广义 live 入口；只做主意图识别、固定模板选择、任务移交 | 新增后承接模糊医学内容请求；仓库文档只保留设计说明，不作为执行入口 |
| 医学写作类目层 | `C:\Users\96138\.config\opencode\skills\medical-writing\SKILL.md` | `medical_kb_system/skills/medical_article_generator_v2.md`、本方案文档 | 唯一医学写作类目 live 入口 | 保留 `medical-writing` 名称；不再承担广义总入口职责 |
| 观点子模块层 | `C:\Users\96138\.config\opencode\skills\medical-opinion-generator\SKILL.md` | `medical_kb_system/skills/medical-opinion-generator.md` | 独立观点子模块入口 | 保留为 live 子模块；由广义触发层或显式直调进入 |
| 审核子模块层 | `C:\Users\96138\.config\opencode\skills\medical-article-reviewer\SKILL.md` | `medical_kb_system/skills/medical-article-reviewer.md` | 独立审核子模块入口 | 保留为 live 子模块；本轮同步进入原型化改造 |
| 查询子模块层 | `C:\Users\96138\.config\opencode\skills\medical-database-query\SKILL.md` | `medical_kb_system/skills/medical-database-query.md` | 独立查询子模块入口 | 保留为 live 子模块；本轮先冻结并定义兼容契约，不列入首批重写 |

这张表同时锁定 3 个决定：

- `medical-writing` 最终保留为“医学写作类目层”，不升级为广义 live 入口。
- 新增的广义触发 skill 正式命名为 `medical-content-router`，路径固定在 OpenCode live skill 目录。
- 仓库内 V1 旧 skill 与 V2 方案文档只保留为规则基线和设计说明，不再作为执行入口。

### 3.4 控制面边界

本方案只定义医学 skill 的分层、调用契约和替代关系，不重写项目控制面。

以下对象继续以 `project_standards/must_read/system_operations.md` 与 `spec需求文档.md` 为唯一真相源：

- 单阶段单权威
- `已实现 / 已验证 / 已关闭` 状态口径
- 冻结、变更、留档与日志机制
- `workflow-gate` 与项目级正式计划入口

医学 skill 方案只声明“读取并遵守”这些控制面规则，不在本方案内再派生一套同名规则。

### 3.5 广义入口替代与旧触发收缩规则

`medical-content-router` 上线后，必须把“新增 router”与“收缩旧 `medical-writing` 触发边界”视为同一轮原子更新。

硬替代规则如下：

- 所有“模糊 / 组合 / 多步 / 先判断再执行”的医学请求，只允许命中 `medical-content-router`。
- `medical-writing` 只接受两类请求：
  - **显式直写请求**：用户已经明确要直接成稿，且写作目标、素材或下游输入已基本齐备。
  - **模板内下游调用**：由 `medical-content-router` 或既定模板显式转入写作类目。
- 只要 `medical-content-router` 已进入 live，`medical-writing` 就不得继续保留广义组合触发语句作为并列入口。

因此，当前 `medical-writing` live skill 中下面这类旧触发语句都应在 router 上线时同步退役：

- “观点 + 写作 + 审核”
- “从这些证据形成观点 + 写篇文章”
- “查一下这个病的知识 + 写一篇”
- “帮我审核这篇文章 + 顺便改一改”
- “完整链路”
- “生成医学内容”
- “检查合规性”

保留给 `medical-writing` 的触发语义，只能是“明确直写”或“模板内已定稿方向”的写作请求，例如：

- 已经给出素材、受众、原型或大纲，要求直接出稿
- 已有 query / opinion / review 结果，要求进入成稿或改写
- 明确指定按某个原型直接起草

“显式直写”还必须满足操作级阈值，不能只看用户是否说了“写一篇”：

- 只有当当前请求或已继承上下文中，`source_packet / prototype_hint / target_audience / rewrite_target` 至少同时具备 2 项时，才允许直达 `medical-writing`
- 其中 `source_packet` 可以是用户已提供素材、明确证据包，或 query / opinion / review / planning 的结构化结果
- 未达到上述阈值的泛写作请求，一律先进入 `medical-content-router`，例如“写一篇医学文章”“帮我写个医学内容”“写一篇关于 XX 的文章”
- 模板内下游调用不受这条阈值限制，但仍必须显式携带最小交接契约进入写作类目

未同步完成 router 上线和 `medical-writing` 触发收缩前，不得对外宣称“广义入口已唯一化”。

这条唯一化规则还必须覆盖现有 live 子模块的组合请求转交目标，不能只改 `medical-writing` 一处：

- `medical-opinion-generator`、`medical-article-reviewer`、`medical-database-query` 当前如果仍写着“组合请求请调用 `medical-writing`”，就仍然构成隐藏的旧入口回流
- 因此，`medical-content-router` 上线同轮，必须同步更新这 3 个子模块的注意项与转交目标
- 更新后的硬规则应统一为：凡“组合 / 多步 / 模糊 / 先判断再执行”的请求，一律改指向 `medical-content-router`，不再改指向 `medical-writing`
- 只有“单一能力、边界清晰”的请求，才允许继续直接命中对应子模块

### 3.6 会话级控制握手与日志初始化归属

为避免链式模板内重复读取控制面、重复确认、重复建日志，本方案补一条会话级 owner 规则：

- **唯一 owner = 链路首个 live 入口**
  - 若请求先进入 `medical-content-router`，则 router 是本会话唯一控制握手 owner。
  - 若用户绕过 router 直接显式调用某个子模块或 `medical-writing`，则该被直调 skill 是 owner。
- owner 负责且只负责一次：
  - 读取 `系统运行.md` 等控制面文档
  - 初始化本次会话的日志上下文
  - 向用户做一次启动确认
- 下游被模板调用的 skill 只继承上下文，不再重复执行启动握手和日志初始化。

链路内下游 skill 允许做的事只有：

- 消费继承来的控制上下文与日志上下文
- 在既有日志上下文里追加模块级执行记录
- 基于上游 payload 继续完成本模块职责

链路内下游 skill 明确禁止再做：

- 第二次读取控制面并向用户重复确认
- 第二次初始化会话级日志
- 以本模块名义重新建立一套并列启动口径

## 四、建议保留不动的部分

以下内容不应被视为“定式问题”，而应继续保留为硬核底盘：

### 4.1 证据先于判断

这一条必须保留：

- 先有证据
- 再组织判断
- 最后形成写作表达

没有证据映射，不应做强结论。

### 4.2 合规与边界控制

这些都应继续保留为不可放松的规则：

- 适应症边界
- 绝对化用语限制
- 非头对头比较限制
- 数据与来源映射
- 风险与限制提示

### 4.3 语域控制

语域分层仍然有价值，尤其对以下场景：

- 专科医师稿
- 患教稿
- 大众科普稿
- 观点稿

问题不在于有语域，而在于不能让语域本身替代文章原型判断。

### 4.4 反翻译腔与中文书面表达底线

这一层也应保留：

- 避免机器翻译腔
- 避免过口语
- 保持中文医学写作的稳定书面感

## 五、建议调整的部分

### 5.1 把“统一叙事骨架”从默认总线降级为一种原型

当前最需要调整的是这一点：

不要再让所有文章都默认走同一套固定骨架。

建议把原来的统一骨架，改成“文章原型库”中的一种，而不是全部任务的默认答案。

### 5.2 把“一文一主张”改成按文章类型生效

这条适合：

- 观点稿
- 社论稿
- 带鲜明判断的证据解读稿

但不应强行覆盖：

- 多维获益盘点稿
- 综述型稿件
- 机制解释稿
- 系列稿中的结构篇

建议改成：

- 对“观点 / 社论 / 单主题判断稿”默认启用
- 对“综述 / 系列 / 机制 / 患教”降为可选约束

### 5.3 把段落功能块从“每段必绑”改成“可调用工具箱”

功能块仍然有价值，但不应要求每段都像拼积木一样被明确绑定。

更合理的做法是：

- 把它保留成可调用工具箱
- 在不同文章原型里，允许不同段落密度和不同位置分布

### 5.4 审核器不要再只盯一个骨架

审核应从“是否符合单一骨架”转成：

- 这篇文章是否匹配它所属的文章原型
- 结构是否自洽
- 证据和表达是否匹配

否则生成器改了，审核器还会把所有内容拉回旧轨道。

## 六、建议新增的“文章原型层”

建议在医学写作类目内部新增“文章原型”判断。

长期目标仍可覆盖 7 类：

1. **证据解读稿**
2. **临床决策稿**
3. **机制解释稿**
4. **患者教育稿**
5. **系列稿**
6. **综述盘点稿**
7. **观点社论稿**

但实施上不建议一次性把 7 类全部写死。

### 6.1 先做 MVP 三原型

本轮 MVP 目标在这里明确锁定为：**先验证新封装是否真的能把原型差异拉开**，而不是先追求最低实施风险。

因此，第一批继续保留差异最大的 3 类，作为最小可用原型：

1. **证据解读稿**
2. **患者教育稿**
3. **观点社论稿**

原因很直接：

- 这 3 类在目标读者、证据密度、语言风格、结尾动作上差异最大
- 只要这 3 类能明显拉开气口，就足以验证“原型层”确实比单骨架更有效
- 如果这 3 类都拉不开，再一次性上 7 类，只会把“一个硬骨架”变成“7 个硬骨架”

补充说明：

- 如果后续某一阶段把目标切换为“最低风险先落地”，第三个原型再改成 `临床决策稿` 更稳。
- 但这不属于本轮 Phase 1 目标，本稿不在这一轮改动第三原型选型。

### 6.2 每类原型的最小定义要求

每种原型不能只给名字，至少要同时定义 5 件事：

- 适用场景
- 默认结构
- 证据密度要求
- 结尾动作方式
- 审核标准

也就是说，**原型定义和审核标准必须一起出现**，不能先定义原型，后面再补审核。

### 6.3 第一批三原型建议定义方式

第一批 3 类建议先按下面粒度收口：

| 原型 | 适用场景 | 默认结构重点 | 证据密度 | 结尾动作 | 审核重点 |
|------|----------|-------------|----------|----------|----------|
| 证据解读稿 | 新研究、会议数据、关键试验结果解读 | 研究问题 -> 结果 -> 边界 -> 启示 | 高 | 结论 + 风险边界 | 数据准确性、研究边界、证据匹配 |
| 患者教育稿 | 患者、照护者、基层传播 | 问题解释 -> 影响 -> 应对建议 | 中 | 行动建议 + 风险提示 | 术语转译、可理解性、过度承诺 |
| 观点社论稿 | 行业判断、证据立场、价值判断 | 立场 -> 支撑 -> 反例/边界 -> 判断 | 中高 | 立场收束 | 论点单一性、证据闭环、边界透明 |

这样一来：

- 稳定性保留
- 同质化下降
- 审核标准也能按原型切换

## 七、建议新增的“广义触发层”

### 7.1 广义触发层负责什么

广义触发层不直接承担最终成稿，也不应膨胀成第二个总控层。

它的正式角色应被收紧为：**固定路由模板编排器**。

也就是说，它只回答三个问题：

1. 用户当前的主意图是什么？
2. 应该匹配哪一个预定义链路模板？
3. 模板选定后，把任务交给哪个下游类目层或子模块？

它的目标不是取代现有子 skill，而是减少前序匹配成本，并承担最小必要的链式编排。

### 7.2 广义触发层允许做的事

广义触发层只允许做以下 3 件事：

1. **识别主意图**
   - 在“查询 / 观点 / 写作 / 审核 / 系列策划或原型判断”之间识别当前主任务。
2. **选择一个预定义链路模板**
   - 只能从事先定义好的模板中选，不允许临场发明自由工作流。
3. **把任务移交给下游**
   - 向 `medical-writing` 或各子模块传递用户目标、素材上下文和必要输入。

### 7.3 广义触发层明确禁止做的事

为防止它继续膨胀，本方案明确禁止广义触发层做以下事情：

- 定义文章原型，或决定原型内部结构细则
- 定义证据选择规则、证据分级规则、`L3/L4` 查询规则
- 定义审核标准、`P0/P1/P2` 分级口径，或把 `M2` 继续设成全局审核门槛
- 重述或覆盖项目冻结、留档、变更、状态切换与日志规则
- 现场自由拼装超出预定义模板上限的长链路

### 7.4 固定链路模板

当前上游 `medical-writing` 的触发条件里，已经包含组合意图，例如：

- “帮我产出一个医学观点 + 写出来”
- “查一下这个病的知识 + 写一篇”
- “帮我审核这篇文章 + 顺便改一改”

这说明真实请求并不是简单的五选一，而是经常包含：

- 串行请求
- 组合请求
- 先判断再执行的请求

因此，广义触发层不能只是做“分类器”，但它的编排必须被限制在固定模板中。

建议首批只开放下面这些模板：

| 模板名 | 触发场景 | 固定链路 |
|--------|----------|----------|
| `query_only` | 只查知识或证据 | `medical-database-query` |
| `opinion_only` | 只形成观点 | `medical-opinion-generator` |
| `review_only` | 只做审核 | `medical-article-reviewer` |
| `write_direct` | 已有素材，直接成稿 | `medical-writing` |
| `query_to_write` | 先查再写 | `medical-database-query -> medical-writing` |
| `opinion_to_write` | 先形成观点再写 | `medical-opinion-generator -> medical-writing` |
| `review_then_rewrite` | 先审再改 | `medical-article-reviewer -> medical-writing` |
| `planning_to_write` | 先判断原型或系列策划，再决定是否成稿 | `medical-writing(原型判断/策划模式) -> 可选进入成稿模式` |

约束补充：

- 每条模板最多 2 到 3 步，不再往上叠加自由节点。
- 原型判断只能发生在 `medical-writing` 类目层内，不在广义触发层定义。
- 一旦模板选定，后续执行权移交给下游 live skill，不再保留双重解释权。

### 7.5 这层的真实价值

这层加上以后，执行者不需要再频繁临场判断：

- 是该先出观点还是直接写
- 是该先查证据还是直接组织文章
- 是该走系列稿还是单篇稿

但它仍然不是新的控制面，也不是新的文章规则中心。

## 八、建议的最终封装结构与兼容契约

建议把医学内容能力封成下面这套结构：

### A. 广义入口

`medical-content-router` 作为唯一广义 live 入口，负责：

- 识别任务意图
- 选择固定链路模板
- 把任务上下文移交给 `medical-writing` 或各子模块

它不定义任何文章原型、证据规则、审核标准或项目控制面规则。

### B. 医学写作类目

`medical-writing` 作为被验证过、可稳定复用的医学写作类目，内部结构分成：

1. **硬约束层**
   - 证据
   - 合规
   - 语域
   - 中文表达底线

2. **文章原型层**
   - 7 类原型
   - 每类默认骨架不同

3. **写作策略层**
   - 开篇方式
   - 证据组织方式
   - 风险段位置
   - 结尾动作方式

4. **风格变体层**
   - 同一原型下允许不同气口
   - 不改变证据与合规底线
   - `Style-RAG` 作为这一层的主要实现机制保留

### B1. Style-RAG 的归属

`Style-RAG` 不建议删除，也不建议继续悬空。

在新结构里，它更适合明确归到“风格变体层”：

- 不负责决定文章原型
- 不负责替代证据判断
- 不负责定义合规规则
- 只负责在同一原型下，帮助输出不同但受控的语感变体

这样它的边界会比旧体系更清晰：

- 原型层决定“这篇文章该长成什么类型”
- 风格变体层决定“同一类型下，这篇文章怎么说更像某种成熟写法”

并且调用边界要写死：

- **可主动调用者**：`medical-writing` 以及未来按原型拆出的写作子模块
- **只可被动消费者**：`medical-database-query`、`medical-article-reviewer`
- 查询与审核模块只允许消费 `Style-RAG` 产生的风格标签、元信息、语域建议或示例结果，不得各自再定义独立风格检索规则
- 如果未来确实需要跨模块共享主动风格检索能力，应另起“公共服务层”，而不是继续挂在“风格变体层”名下

### B2. M0-M7 到新四层结构的映射

这里的映射是**职责重分配**，不是“旧体系直接废弃”，也不是“把旧名字原样平移到新名字”。

建议映射如下：

| 旧模块 | 主属层 | 次级依赖层 | 说明 |
|--------|--------|--------------|------|
| M0 Persona / Lens | 文章原型层接口 | 硬约束层 | 由原型层拥有“谁对谁说”的定义权，硬约束层消费其受众与语域边界 |
| M1 Thesis & Insight | 文章原型层 | 写作策略层 | 主张强度和数量先由原型决定，再由策略层决定如何展开 |
| M2 Universal Narrative | 文章原型层 | 写作策略层 | 从“全局默认骨架”降为“某一类原型的默认骨架” |
| M3 Strategic Plays | 写作策略层 | 文章原型层 | 继续保留为打法库，但不再反向支配原型分类 |
| M4 Rhetoric Blocks | 写作策略层 | 风格变体层 | 从“每段必绑”改成“可调用工具箱”，具体语感由风格层细化 |
| M5 Compliance | 硬约束层 | 文章原型层 | 仍是强门禁，但不同原型可声明不同的边界提示方式 |
| M6 Evidence Database | 硬约束层 | 文章原型层 | 继续作为事实与证据底盘，原型层只决定证据密度和组织方式 |
| M7 Orchestrator | 类目总线接口（非内容定义层） | 硬约束层 / 文章原型层 / 写作策略层 / 风格变体层 | 继续保留，但只负责编排，不再拥有任何单层内容定义权 |

一句话：

> 不是 M0-M7 被替换，而是 M0-M7 从“单线强编排”重构为“分层可组合”，并且每一块都有更清晰的主属层。

### C. 子模块

保留并继续复用：

- 观点产出
- 文章审核
- 数据查询

必要时再新增：

- 系列策划
- 素材蒸馏
- 证据包整理

### C1. 查询与审核的兼容契约

本轮对 `medical-database-query` 与 `medical-article-reviewer` 采用“先锁兼容，再分批重构”的策略。

1. **`medical-database-query`**
   - 本轮不进入首批结构重写，但必须纳入冻结矩阵。
   - 继续保留 `query_type / question / results / total_found / returned` 的结构化输出契约。
   - 继续保留 `L3/L4` 分层、来源标注、质量门与对下游观点链的结构化对接。
   - 允许新增可选元信息，如 `recommended_register`、`style_metadata`，但不允许内建独立 `Style-RAG` 检索流程。

2. **`medical-article-reviewer`**
   - 本轮进入重构范围，但以“原型审核标准上线即替换旧通用规则”的方式推进。
   - 保留专业性、合规性、可读性这些通用底线。
   - 逐步取消把 `M2` 单骨架当作所有原型总门槛的做法。

### C2. 链式模板最小交接契约

为避免不同执行者在实现时各自发明 payload，所有跨模块模板都至少继承同一组公共上下文：

- `route_template_id`
- `handoff_source`
- `control_context`
- `log_context`

其中：

- `control_context` 用于表明会话级控制握手已完成，不再重复启动握手。
- `log_context` 用于沿用同一会话日志，不再重复初始化。

`medical-writing` 作为下游写作类目，统一接收：

- `task_goal`
- `source_packet`
- `constraints_context`
- 至少一个写作落点信号：`prototype_hint / rewrite_target / target_audience`

在这个统一入口之上，各模板再补最小必要字段：

| 模板 | 上游 -> 下游 | 最小交接字段 |
|------|--------------|--------------|
| `query_to_write` | `medical-database-query -> medical-writing` | `question / results / source_summary / recommended_register` |
| `opinion_to_write` | `medical-opinion-generator -> medical-writing` | `thesis / support_points / boundaries / evidence_mapping` |
| `review_then_rewrite` | `medical-article-reviewer -> medical-writing` | `draft_ref_or_text / review_findings / severity_summary / rewrite_target` |
| `planning_to_write` | `medical-writing(策划模式) -> medical-writing(成稿模式)` | `prototype_recommendation / planning_rationale / required_inputs / write_decision` |

约束说明：

- 这里只固定最小交接契约，不在本方案内扩成完整 schema。
- 各模板可以增加字段，但不得删掉最小字段，避免同一模板出现多套互不兼容的 payload。
- `planning_to_write` 的策划输出从本稿起视为显式契约，而不是实现时再临场决定。

## 九、实施顺序

建议不要一步把所有 skill 全改掉，而是按下面顺序推进：

### 第一步：锁定 live 名称、入口归属与收缩规则

先按第 3.3 节把 live 名称、入口归属与旧触发收缩规则钉死，再进入实施设计：

- 固定 `medical-content-router` 作为唯一广义 live 名称与入口归属
- 保留 `medical-writing` 为医学写作类目层 live 入口
- 写死 `medical-writing` 的旧广义触发收缩规则与“显式直写”判定阈值
- 仓库文档、V1 旧 skill 和 V2 `README` 只作设计说明与基线，不作为临时第二入口

这一阶段必须同轮完成两件事：

- 固定 `medical-content-router` 的 live 名称、路径与唯一入口归属
- 固定 `medical-writing` 的旧广义触发退役清单，只保留显式直写与模板内下游调用，并写死显式直写阈值
- 固定 3 个 live 子模块的 redirect 迁移规则：`medical-opinion-generator / medical-article-reviewer / medical-database-query` 遇到组合请求时，统一改指向 `medical-content-router`

如果只做前者、不做后者，就仍然属于双入口并存，不得宣称入口已唯一化。

### 第二步：冻结四类能力的基线矩阵

冻结对象不再只盯 3 个旧核心，而是扩成“写作 / 观点 / 审核 / 查询”四类能力：

| 能力 | live 上游基线 | 仓库旧规则基线 | 当前真实冲突源 | 本轮范围 | 处理原则 |

### 第三步：重写医学写作类目定义

重写 `medical-writing` 的说明，明确：

- 哪些是硬约束
- 哪些是文章原型
- 哪些是可变策略
- 哪些只属于风格变体

这一阶段不要一次性定义全部 7 类原型，而是先完成：

- 证据解读稿
- 患者教育稿
- 观点社论稿

并且每定义 1 类原型，必须同步交付该原型的审核标准。

### 第四步：实现广义触发层文件、模板逻辑与 handoff 接线

在第 1 步已经锁定名称与入口归属的前提下，再实际实现 `medical-content-router` 文件、固定模板逻辑和到下游的 handoff 接线。

这一阶段的落地动作是：

- 创建或改造 `medical-content-router` live skill 文件
- 把更模糊的需求接入 router，再把明确写作任务路由到医学写作类目或其他子模块
- 按既定模板把 handoff 契约接到 `medical-writing`、`medical-opinion-generator`、`medical-article-reviewer`、`medical-database-query`

router 上线时的 rollout checklist 还必须写死：

- 同步收缩 `medical-writing` 的旧广义触发语句与显式直写阈值
- 同步把 `medical-opinion-generator`、`medical-article-reviewer`、`medical-database-query` 中“组合请求请调用 `medical-writing`”改成“组合请求请调用 `medical-content-router`”
- 同步复核所有 live 医学 skill，确认不再存在任何把组合请求直接导回 `medical-writing` 的旧说明

这一阶段必须写死：

- 它只做主意图识别
- 它只选固定链路模板
- 它只负责把任务移交给下游
- 它不定义原型、证据、审核或项目控制面规则

并且在模板执行层面同步落地：

- 由链路首个 live 入口承担一次性的控制握手和日志初始化
- 下游被调用 skill 只继承 `control_context` 与 `log_context`，不再重复启动握手

### 第五步：调整审核器，并写死替代关系

审核器改造不能再只写“按原型审核”，还必须把旧规则何时退役写硬：

| 旧 reviewer 规则 | 新状态 | 替代关系 |
|------------------|--------|----------|
| 专业性审查 | 继续保留为全原型通用底线 | 仍对所有原型生效 |
| 合规性审查 | 继续保留为全原型通用底线 | 仍对所有原型生效 |
| 可读性审查 / Register / 翻译腔 | 继续保留为全原型通用底线 | 仍对所有原型生效 |
| `M2骨架完整性` | 降级为旧单骨架模型专用规则 | 当某原型审核标准上线后，不再作为该原型的通用门槛；仅对显式沿用旧骨架的稿件生效 |
| 论点单一性 | 从全局规则改为原型范围规则 | 默认只对 `观点社论稿` 与单主题判断型稿件强制生效 |
| 逻辑连贯性 / 详略得当 | 保留，但评价对象改写 | 以后按“是否符合该原型结构”判断，不再按单一 `M2` 判断 |

执行原则：

- 某原型审核标准一旦上线，凡与之冲突的旧 reviewer 项自动降为“旧体系兼容规则”或“参考项”。
- 不允许一边上线新原型审核标准，一边继续把旧 `M2骨架完整性` 当总门槛使用。

### 第六步：锁定查询兼容契约与风格边界

在不首批重写 `medical-database-query` 的前提下，至少同步落地两件事：

- 写死 query 的结构化输出契约与 `L3/L4` 分层边界
- 写死 `Style-RAG` 的调用边界：写作侧可主动调用，query/review 只能消费结果或元信息

同一步补齐链式模板最小交接契约，至少固定：

- `query_to_write`
- `opinion_to_write`
- `review_then_rewrite`
- `planning_to_write`

### 第七步：用 3 类真实文章做试跑

建议至少拿这三类做试跑：

- 证据解读稿
- 患教稿
- 观点社论稿

验收重点不是“写得像不像旧模板”，而是同时验证：

- 入口是否唯一，没有双入口并存
- 原型之间是否真正拉开结构和语感差异
- 审核器是否不再被旧 `M2` 总门槛绑回去
- query 与 opinion 子模块是否仍能按兼容契约稳定输出

---

## 十、结论

这次调研后，我的判断是：

1. `medical-writing` 最终保留为“医学写作类目层”，不再兼任广义总入口。
2. 新增 `medical-content-router` 作为唯一广义 live 入口，负责主意图识别、固定模板选择和任务移交。
3. 当前 live skill 并不全是薄壳，尤其 reviewer/query 仍带实质规则；因此冻结矩阵必须覆盖“写作 / 观点 / 审核 / 查询”四类能力。
4. 广义触发层必须被收紧为“固定路由模板编排器”，不能再膨胀成第二个总控层。
5. 审核器改造必须和原型定义同步推进，并写死旧 reviewer 规则的替代关系，尤其是 `M2骨架完整性` 不再作为全原型总门槛。
6. `Style-RAG` 继续保留在风格变体层，但主动调用权只留给写作侧；query/review 只能消费结果或元信息。
7. Phase 1 的 MVP 目标明确是“验证原型差异是否成立”，因此继续保留 `证据解读稿 / 患者教育稿 / 观点社论稿` 这组组合。
8. 项目冻结、留档、变更、状态切换与日志机制继续由 `系统运行.md` 和 `spec需求文档.md` 承接，本方案不再复制控制面。
9. `medical-content-router` 上线必须与 `medical-writing` 旧广义触发语句收缩、“显式直写”阈值生效，以及 3 个 live 子模块 redirect 迁移同轮完成，否则仍属于双入口并存。
10. 链式模板中的控制面读取、启动确认与日志初始化，只能由链路首个 live 入口执行一次；下游模块只继承上下文。
11. `planning_to_write` 以及其他跨模块模板，从本稿起都必须遵守最小交接契约，不再允许实现时临场发明 payload。
