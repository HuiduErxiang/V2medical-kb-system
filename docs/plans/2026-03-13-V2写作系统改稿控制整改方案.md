# V2 写作系统改稿控制整改方案

> **日期**: 2026-03-13
> **性质**: 整改方案 / 待复核
> **当前状态**: 未执行
> **适用范围**: `medical_kb_system_v2` 写作链路中与长文改稿、去重、增补、深化相关的控制层问题

---

## 1. 问题定义

当前 `V2` 写作系统已经对入口唯一化、Router 分流、单次握手、handoff 契约和三原型做过多轮整改，但在长文改稿场景仍反复出现以下问题：

1. 让系统“增补内容”时，容易通过过渡句、抽象总结和同义复述来扩字数，形成灌水。
2. 让系统“修正重复段落”时，容易直接删段，导致字数和信息量一起下降。
3. 让系统“增加深度”时，容易写成更宏观的升华，而不是更具体的机制、证据、对比和边界。
4. 用户已经明确锁定的内容，例如引言不改、结构不改、禁止写某类内容，仍可能在后续轮次中被改动或被重新引入。

这说明当前问题已经不再是“如何进入写作系统”，而是“进入写作系统后，如何做受控改稿”。

---

## 2. 根因判断

### 2.1 已整改的层

当前系统已经较清楚地覆盖了以下层面：

- Router 入口分流
- `write_direct` 直写阈值
- 下游继承 `control_context` / `log_context`
- `query_to_write / opinion_to_write / review_then_rewrite / planning_to_write` 的基础交接字段
- 三原型的结构定义

### 2.2 尚未整改的层

当前尚未形成稳定机制的，是**改稿控制层**，具体缺口有 4 类：

1. **缺少冻结 brief 机制**
   - 系统没有要求“当前轮唯一有效 brief”先被冻结，再进入写作或改写。
   - 项目执行卡、需求文档、项目日志中的冲突约束会被同时送入模型，导致模型在冲突条件下折中输出。

2. **缺少锁段改写机制**
   - 当前写作输入只有 `source_packet / prototype_hint / rewrite_target / target_audience` 等高层字段。
   - 没有 `locked_spans`、`must_keep_sections`、`must_not_touch` 之类的段落级控制。
   - 结果是用户说“只改 Part 2，不动引言”，系统仍可能在整稿层重写。

3. **缺少去重与增补的模式分离**
   - 当前系统把“增补”“去重”“深化”都当作一个笼统的 `rewrite_target`。
   - 模型拿到泛化改写目标时，会自行决定删哪里、补哪里、升华哪里。
   - 没有“增补只能增加有效信息”“去重必须信息守恒”“深化只能增加论证层级”的硬约束。

4. **缺少 reviewer 到 writer 的精细 patch 契约**
   - reviewer 当前只输出 `review_findings / severity_summary / rewrite_target`。
   - 这不足以指导 writer 做段落级修正。
   - writer 因此只能做整稿重求解，而不是基于差异的局部 patch。

---

## 3. 整改目标

整改后，系统至少要满足以下目标：

1. 用户明确锁定的内容不得被后续轮次改坏。
2. “增补”“去重”“深化”必须进入不同的受控模式，不再共用一个泛化改写入口。
3. reviewer 输出必须能直接指导 writer 做段落级 patch，而不是只给抽象问题清单。
4. 写作系统必须知道哪些信息是“不能删的内容”，哪些是“允许压缩的表达”，哪些是“缺失的证据/机制/边界”。
5. 项目级唯一 brief 必须先冻结，后续轮次不再并行服从多个控制文档。

---

## 4. 整改方案

### 4.1 新增“冻结 brief”步骤

在任何长文改稿前，先生成一份**当前轮冻结 brief**，作为唯一执行真相源。

冻结 brief 最少包含：

- `project_id`
- `audience`
- `article_goal`
- `prototype_hint`
- `locked_sections`
- `must_keep_points`
- `must_delete_points`
- `forbidden_topics`
- `word_count_target`
- `evidence_gaps`
- `current_source_of_truth`

规则：

1. 未冻结 brief 前，不允许进入成稿或改稿。
2. 若项目执行卡、需求文档、项目日志冲突，必须先解决冲突，再生成冻结 brief。
3. 一轮冻结 brief 生效后，本轮只服从这份 brief，不再并行服从其他控制文档。

### 4.2 把改稿拆成 3 个受控模式

#### 模式 A：增补模式 `expand_patch`

只允许新增以下内容：

- 数据
- 机制解释
- 对比基准
- 案例或场景
- 适用边界
- 推导链条

禁止新增：

- 空泛过渡语
- 抽象口号
- 不产生新信息的同义复述

#### 模式 B：去重模式 `dedupe_patch`

目标不是“删字”，而是“压缩重复表达并保留信息量”。

必须输出：

- 删除了哪些重复点
- 保留到哪一段
- 哪些信息被合并而不是被删除
- 改前/改后字数变化
- 信息守恒判断

#### 模式 C：深化模式 `deepen_patch`

深化必须先指定维度，只能选择以下一种或多种：

- 机制深化
- 证据深化
- 对比深化
- 边界深化
- 商业/临床推导深化

禁止把“深化”理解为：

- 更宏大的总结
- 更抽象的判断
- 更多修辞

### 4.3 扩展 reviewer 输出契约

当前 reviewer 输出过于抽象，应新增一个面向 writer 的 patch 契约。

建议新增字段：

```json
{
  "rewrite_mode": "expand_patch | dedupe_patch | deepen_patch | mixed_patch",
  "locked_spans": [
    { "section_id": "intro", "action": "keep_verbatim" }
  ],
  "must_keep_claims": [
    "PACC突变领域目前无获批药物"
  ],
  "must_delete_or_compress": [
    {
      "issue_id": "dup-001",
      "location": "Part 2",
      "instruction": "压缩为一句总结，不再重复引言中的 exon20ins 先发优势"
    }
  ],
  "expand_targets": [
    {
      "location": "Part 1",
      "dimension": "mechanism",
      "instruction": "补充空间位阻为何导致传统TKI难以稳定结合"
    }
  ],
  "evidence_gap_list": [
    {
      "location": "Part 1",
      "missing_item": "ELCC官方 ORR/PFS 数据",
      "status": "pending"
    }
  ],
  "style_guard": {
    "forbidden_phrases": ["更深一层来看", "源头创新不是口号"],
    "tone_requirement": "冷静洞察，不喊口号"
  }
}
```

这样 reviewer 不再只说“有重复”“太干瘪”“有 AI 味”，而是给出可执行 patch。

### 4.4 扩展 medical-writing 接收字段

`medical-writing` 不应只接收高层落点信号，还应能接收改稿控制字段。

建议新增：

- `rewrite_mode`
- `locked_spans`
- `must_keep_claims`
- `must_delete_or_compress`
- `expand_targets`
- `dedupe_map`
- `evidence_gap_list`
- `style_guard`
- `word_budget`
- `section_budget`

其中：

- `word_budget`：限制本轮允许增减的总字数范围
- `section_budget`：限制各 section 允许增减的字数范围

这样可以避免“去 AI 味直接少 500 字”“增补深度直接灌水 600 字”。

### 4.5 增加“改稿前校验”

进入改稿前必须先做 5 项检查：

1. 是否已有唯一冻结 brief
2. 是否明确本轮改稿模式
3. 是否列出锁定段落
4. 是否列出必须保留的信息点
5. 是否列出待补证据缺口

任一项缺失：

- 不直接改稿
- 先回到策划/冻结 brief 步骤

---

## 5. 实施顺序

建议分 3 个小阶段做，不要一次性大改：

### Phase A：控制协议先落文档

目标：

- 先把冻结 brief、改稿模式、patch 契约写成文档规范
- 暂时不改动系统整体路由逻辑

产出：

- `改稿控制协议` 文档
- `冻结 brief 模板`
- `reviewer -> writer patch schema`

### Phase B：改造 reviewer 输出

目标：

- 让 reviewer 从“问题清单”升级为“问题清单 + patch 指令”

重点：

- 新增 `rewrite_mode`
- 新增 `locked_spans`
- 新增 `must_delete_or_compress`
- 新增 `expand_targets`

### Phase C：改造 writer 接收与执行

目标：

- 让 writer 从“整稿改写”升级为“受控 patch 执行”

重点：

- 支持 3 类改稿模式
- 支持段落锁定
- 支持字数预算
- 支持信息守恒检查

---

## 6. 验收标准

整改是否成功，至少看这 6 条：

1. 用户明确要求“不改引言”时，引言在改稿后保持不变。
2. 用户要求“补充深度”时，新增内容以机制、证据、对比、边界为主，而不是抽象升华。
3. 用户要求“去重”时，字数可以变化，但关键信息点不丢。
4. reviewer 输出可以直接指导 writer 做局部修补，而不是只能给泛化意见。
5. 项目执行时，系统能明确指出“当前唯一冻结 brief”。
6. 同一项目经过多轮修改后，控制面不会再次出现多份文档互相打架。
