> **性质**: 静态规范文档，不随对话更新  
> **更新触发**: 仅当规范本身需要调整时更新  
> **最后更新**: 2026-03-10

---

## 第一部分：文档通用格式

### 1.1 文件命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 核心规范文档 | `NN_文档名.md` | `00_project_standards.md` |
| 子文件夹内文档 | 直接命名 | `交接文档.md` |
| 归档文档 | `原名_归档日期.md` | `旧规范_20260305.md` |

**编号规则**:
- `00` - 静态规范（总纲）
- `01-04` - 动态文档（随对话更新）
- `05+` - 扩展规范（如书写规范、接口规范等）

### 1.2 文档头部元数据

所有文档必须在开头包含以下元数据（使用引用格式）：

```markdown
> **文档位置**: `相对路径/文档名.md`  
> **性质**: 静态/动态文档  
> **更新触发**: 什么情况下更新  
> **最后更新**: YYYY-MM-DD
```

### 1.3 标题层级

**强制要求**:
- 使用 CommonMark 标准
- 标题前后必须有空行
- 从 H1 (`#`) 开始，依次降级

### 1.4 表格格式

| 列对齐 | 说明 |
|:-------|:-----|
| 左对齐 | 默认，文本内容 |
| 右对齐 | 数值型数据 |
| 居中对齐 | 状态标识 |

### 1.5 列表格式

**无序列表**: 使用 `-` 而非 `*`

**有序列表**: 使用 `1.` `2.` `3.`

**嵌套列表**: 使用 2 个空格缩进

## 第二部分：project_standards文档特殊要求

### 2.1 铁律文档格式（R-XXX）

所有规则必须使用统一格式：

```markdown
| **R-XXX** | **规则名称** | 规则说明 |
```

### 2.2 版本记录格式

所有动态文档必须在文末包含版本记录：

```markdown
| 日期 | 版本 | 变更内容 | 作者 |
```

### 2.3 待办事项格式

使用优先级标识：P0（紧急）> P1（重要）> P2（一般）

### 2.4 项目日志格式

职责边界：
- 本文件只定义项目日志的格式、命名和字段排序
- 何时必须写日志、何时必须触发变更记录，见 [必看/系统运行.md](./必看/系统运行.md)
- 项目写作任务的执行结构，见 [06_项目写作任务模板.md](./06_项目写作任务模板.md)

项目日志固定字段建议：
- 本轮输入
- 已确认规则
- 新增问题
- 已修复项
- 未决事项
- 下一步动作

如仍使用主题簇形式，至少必须包含：
- 主题
- 用户目标
- 关键决策
- 交付物
- 下一步计划

项目日志命名建议：
- `YYYYMMDD_HHMMSS_主题.md`
- 如为单独变更记录，可使用 `YYYYMMDD_HHMMSS_变更记录.md`

## 第三部分：路径引用规范

### 3.1 相对路径规则

**同一目录**: `./文件名.md` 或 `文件名.md`

**子目录**: `./子目录/文件名.md`

**父目录**: `../文件名.md`

### 3.2 路径检查清单

创建文档链接前必须确认：
- [ ] 目标文件存在
- [ ] 路径相对位置正确
- [ ] 中文路径无编码问题
- [ ] 特殊字符（如 emoji）已正确处理

## 第四部分：文档维护最佳实践

### 4.1 单一更新原则

**正文原地修改，更新记录追加**

### 4.2 去重检查

更新文档后必须检查：无重复段落、无矛盾陈述、无过时信息

### 4.3 格式纯净

禁止行为：表格行首添加前缀字符、使用非标准 Markdown 语法、混用不同风格的列表符号

### 4.4 文档长度控制

**建议上限**:
- 核心规范文档：≤ 100 行
- 项目日志：≤ 200 行（通过主题簇管理）
- 版本迭代：≤ 50 行（保留最近 10 个版本）
- 交接文档：≤ 150 行
