# Medical Article Reviewer

## 定位

本 Skill 是医学文章**审核子模块**，负责对医学内容进行多维度审查。

审核标准根据文章原型动态切换，不再使用单一骨架作为全原型总门槛。

---

## Triggers

当用户**仅需要文章审核**，且**不涉及观点产出或写作生成**时，加载本技能：

- "帮我审核一下这篇文章"
- "审稿"
- "检查这篇文章有没有问题"
- "这篇文章合规吗"
- "看看这篇文章质量怎么样"
- "帮我把把关"
- "审查"
- "单独调用审核"

**注意**：如果用户同时要求"写文章"、"产出观点"、"生成完整内容"，请调用 `medical-content-router` skill。

---

## 审核维度

审核时必须覆盖以下维度，分为**通用底线**和**原型范围规则**两层。

### 通用底线（所有原型强制）

以下审查项对所有文章原型强制生效，不可跳过：

| 维度 | 审查内容 | 问题级别 |
|------|----------|----------|
| **专业性审查** | 数据准确性、指南一致性、机制逻辑 | P0/P1 |
| **合规性审查** | 适应症边界、绝对化用语、非头对头比较限制 | P0 |
| **可读性审查** | Register 匹配、翻译腔、长难句 | P1/P2 |

**P0 级问题**：事实错误、合规红线、证据缺失 → 必须修改
**P1 级问题**：逻辑跳跃、详略不当 → 建议修改
**P2 级问题**：表达优化 → 可选修改

---

### 原型范围规则

根据文章原型，追加以下审核重点：

#### 证据解读稿

| 审核重点 | 说明 | 强制级别 |
|----------|------|----------|
| 数据准确性 | 数值、百分比、P值等数据是否准确引用 | P0 |
| 研究边界 | 是否明确研究局限、适用人群、对比基准 | P1 |
| 证据匹配 | 结论是否与证据等级匹配，是否过度推断 | P1 |
| 结构完整性 | 是否包含：研究问题 → 结果 → 边界 → 启示 | P1 |

#### 患者教育稿

| 审核重点 | 说明 | 强制级别 |
|----------|------|----------|
| 术语转译 | 专业术语是否转化为患者可理解的表达 | P1 |
| 可理解性 | 是否使用日常语言，避免术语堆砌 | P1 |
| 过度承诺 | 是否存在"治愈"、"根治"等过度承诺 | P0 |
| 行动建议 | 是否提供具体可执行的建议 | P1 |
| 风险提示 | 是否包含"不确定时咨询医生"等提示 | P1 |

#### 观点社论稿

| 审核重点 | 说明 | 强制级别 |
|----------|------|----------|
| 论点单一性 | 全文是否围绕一个核心论点展开 | P0 |
| 证据闭环 | 每个支撑点是否有证据映射 | P1 |
| 边界透明 | 是否声明主张的适用范围与局限 | P1 |
| 结构完整性 | 是否包含：立场 → 支撑 → 反例/边界 → 判断 | P1 |

---

### 旧骨架规则说明

以下规则已从"全原型总门槛"降级：

| 旧规则 | 新状态 | 替代关系 |
|--------|--------|----------|
| `M2 骨架完整性` | 降级为旧单骨架模型专用规则 | 仅当文章未指定原型时参考；原型审核标准上线后，不再作为该原型的通用门槛 |
| `论点单一性` | 改为原型范围规则 | 默认只对 `观点社论稿` 强制生效 |
| `逻辑连贯性/详略得当` | 评价对象改写 | 按"是否符合该原型结构"判断，不再按单一 M2 判断 |

**执行原则**：
- 某原型审核标准一旦上线，凡与之冲突的旧规则自动降为"旧体系兼容规则"
- 不允许一边上线新原型审核标准，一边继续把旧 M2 骨架当总门槛使用

---

### 原型感知入口

当接收到 handoff 包含 `prototype_hint` 时：
1. 加载对应原型的审核重点
2. 将原型结构作为骨架验收标准
3. 不再使用旧 M2 骨架作为总门槛

当无 `prototype_hint` 时：
1. 只执行通用底线审查
2. 不强制原型结构验收
3. 可参考旧 M2 骨架作为兼容规则

## 控制握手（双模式）

**本 Skill 可能以两种模式被调用**：

### 模式一：单独直接调用

当用户直接触发本 Skill（不经过 `medical-content-router`）时，执行完整控制握手：

**第一步：读取运行时规则**

立即读取以下文件：
```
D:/汇度编辑部1/写作知识库/项目规范/必看/系统运行.md
```

必须确认已读并理解：
- 铁律与约束（SYS-001 ~ SYS-016）
- `workflow-gate` 入口与触发时机
- 文档地图
- 项目日志机制

**第二步：按当前规则创建日志并确认口径**

- 按 DOC-017 执行日志分层保存：
  - 能明确归属到独立项目时，优先写入项目目录下的 `项目日志/`
  - 无法归属到具体项目时，写入 `写作项目运行日志/` 作为摘要索引
- 向用户确认："已读取系统运行并按当前日志机制准备记录，确认理解 SYS-001~016 与关键 DOC 规则。"

### 模式二：作为 Router 下游调用

当由 `medical-content-router` 通过 `review_only` 或 `review_then_rewrite` 模板调用时，**只继承上下文，不重复握手**：

**接收的 Handoff 字段**：
```json
{
  "route_template_id": "review_only | review_then_rewrite",
  "handoff_source": "medical-content-router",
  "control_context": { "已读取控制面": true, "owner": "medical-content-router" },
  "log_context": { "日志路径": "...", "会话ID": "..." }
}
```

**禁止再做**：
- ❌ 第二次读取控制面文档
- ❌ 第二次初始化日志
- ❌ 重复启动确认

**只允许**：
- ✅ 继承 `control_context` 和 `log_context`
- ✅ 在既有日志上下文里追加模块级执行记录
- ✅ 基于上游 payload 继续完成审核任务

## 工作流

### 单独直接调用模式
```
用户直接触发本 Skill
    ↓
读取系统运行.md（铁律、workflow-gate、文档地图、项目日志机制）
    ↓
按 DOC-017 准备日志记录位置
    ↓
向用户确认已理解规则
    ↓
获取待审核文章 + 上下文（产品、受众、Register、prototype_hint）
    ↓
判定原型（从 handoff 或文章内容推断）
    ↓
执行通用底线审查
    ↓
执行原型范围审查
    ↓
按 P0 / P1 / P2 分级输出审核报告
    ↓
交付审核结果
```

### Router 下游调用模式
```
medical-content-router 调用本 Skill
    ↓
继承 control_context 和 log_context
    ↓
基于 handoff payload 获取待审稿件
    ↓
判定原型（从 handoff 或文章内容推断）
    ↓
执行通用底线审查 + 原型范围审查
    ↓
按 P0 / P1 / P2 分级输出审核报告
    ↓
交付审核结果（含结构化输出字段）
```

## 输出给下游的结构化字段

当本 Skill 的结果需要传递给 `medical-writing` 进行改写时，必须输出以下字段（`review_then_rewrite` 模板契约）。

**完整输出字段定义参见**：`medical_kb_system_v2/docs/contracts/改稿控制协议_v1.md`

### 基础字段（原有）

```json
{
  "draft_ref_or_text": "待审稿件引用或文本",
  "review_findings": {
    "P0_issues": [ "问题1", "问题2" ],
    "P1_issues": [ "问题3" ],
    "P2_issues": [ "问题4" ]
  },
  "severity_summary": { "P0": 0, "P1": 2, "P2": 3 }
}
```

### 新增 Patch 控制字段

审核完成后，必须输出以下 patch 控制字段，指导 writer 做受控改稿：

```json
{
  "rewrite_mode": "expand_patch | dedupe_patch | deepen_patch | mixed_patch",
  
  "locked_spans": [
    { "section_id": "intro", "action": "keep_verbatim", "reason": "用户明确要求不改引言" }
  ],
  
  "must_keep_claims": [
    "PACC突变领域目前无获批药物",
    "exon20ins 是 EGFR 罕见突变中先发优势最明确的亚型"
  ],
  
  "must_delete_or_compress": [
    {
      "issue_id": "dup-001",
      "location": "Part 2 第3段",
      "instruction": "压缩为一句总结，不再重复引言中的 exon20ins 先发优势，保留关键数据",
      "preserve_info": ["ORR 60%", "PFS 9.7个月"]
    }
  ],
  
  "expand_targets": [
    {
      "location": "Part 1",
      "dimension": "mechanism",
      "instruction": "补充空间位阻为何导致传统TKI难以稳定结合",
      "forbidden_additions": ["空泛过渡语", "抽象口号"]
    }
  ],
  
  "evidence_gap_list": [
    {
      "location": "Part 1",
      "missing_item": "ELCC官方 ORR/PFS 数据",
      "status": "pending",
      "action_if_missing": "标注'数据待补充'或使用可替代数据源"
    }
  ],
  
  "style_guard": {
    "forbidden_phrases": ["更深一层来看", "源头创新不是口号", "颠覆性突破"],
    "tone_requirement": "冷静洞察，不喊口号",
    "register": "R3"
  },
  
  "word_budget": {
    "total_delta": { "min": -100, "max": +200 },
    "section_budgets": {
      "intro": { "action": "locked", "delta": 0 },
      "part1": { "action": "expand", "delta": "+50~+100" },
      "part2": { "action": "compress", "delta": "-30~-50" }
    }
  }
}
```

### 字段说明

| 字段 | 用途 | 必需 | 说明 |
|------|------|------|------|
| `rewrite_mode` | 改稿模式 | 是 | 必须从三种模式中选择：expand_patch / dedupe_patch / deepen_patch / mixed_patch |
| `locked_spans` | 锁定段落 | 有锁定时 | 标记不得改动的段落，writer 必须遵守 |
| `must_keep_claims` | 必留信息点 | 是 | 核心信息点列表，改稿后必须保留 |
| `must_delete_or_compress` | 必删/必压内容 | 有问题时 | 指明需要删除或压缩的内容，以及保留的信息 |
| `expand_targets` | 增补目标 | expand 模式时 | 指明增补维度、位置和约束 |
| `evidence_gap_list` | 证据缺口 | 有缺口时 | 待补证据列表和处理建议 |
| `style_guard` | 风格约束 | 否 | 禁用词、语调要求、Register |
| `word_budget` | 字数预算 | 否 | 总字数预算和分段预算 |

### 输出原则

1. **问题必须可执行** — 不只说"有重复""太干瘪""有 AI 味"，必须翻译成具体的 patch 指令
2. **锁定必须明确** — 哪些段落不能动，必须清楚标注
3. **信息必须守恒** — 去重时说明删了什么、保留了什么
4. **增补必须受控** — 增补时指定维度，禁止灌水

## 改稿模式判定规则

根据审核发现的问题类型，自动判定改稿模式：

| 问题类型 | 推荐模式 | 判定依据 |
|----------|----------|----------|
| 内容重复、冗余表达 | `dedupe_patch` | 多处表达相同信息 |
| 证据缺失、论证单薄 | `expand_patch` | 缺少数据支撑、机制解释 |
| 分析浅显、缺乏深度 | `deepen_patch` | 只有结论无推导过程 |
| 混合问题 | `mixed_patch` | 同时存在以上多种问题 |

**注意**：`mixed_patch` 时，必须在 `must_delete_or_compress` 和 `expand_targets` 中分别标注各段落的具体模式。
